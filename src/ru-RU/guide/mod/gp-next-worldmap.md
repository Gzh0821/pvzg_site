# Карта мира и `gpn-worldmap.json5`

> [!warning]
> Эта возможность пока относится к вкладке **Experimental** в GP-Next.  
> Перед использованием включите `worldmap-json` на внутриигровой странице **Experimental**.

Если вы хотите менять узлы, линии, ветки и порядок наградных точек, которые игрок реально видит на карте мира, обычного редактирования `WorldmapFeatures` недостаточно. Нужно использовать:

```text
jsons/worldmap/gpn-worldmap.json5
```

Это сильно отличается от обычного дата-патча:

- `features/WorldmapFeatures.json5`: меняет сырые данные мира, такие как `LEVELS`, `PLANTS`, `STARTINGLEVELS`, `EPIC_TARGET`
- `worldmap/gpn-worldmap.json5`: меняет runtime-граф островов, который реально отображается на карте

Проще говоря:

- если вы хотите изменить, какие уровни, растения или вступление есть у мира, сначала смотрите `WorldmapFeatures`
- если вы хотите вставить растение после первого уровня, перенести подарочную коробку на mainline или перестроить ветки, используйте `gpn-worldmap`

## Минимальный файл

Текущий формат требует корневое поле `apiVersion`, и сейчас принимается только версия `1`:

```json
{
  "apiVersion": 1,
  "worlds": {
    "egypt": {
      "map": {
        "mode": "replace",
        "mainline": []
      }
    }
  }
}
```

Предпочтительное имя файла:

- `gpn-worldmap.json5`

Если вам нужен обычный JSON вместо JSON5, можно также использовать:

- `gpn-worldmap.json`

## Куда его класть

Внутри datapack структура обычно выглядит так:

```text
MyWorldmapPack/
├── pack.json
└── jsons/
    ├── features/
    ├── levels/
    └── worldmap/
        └── gpn-worldmap.json5
```

## Рекомендуемая структура

Сейчас для `replace` рекомендуется такая форма:

- `mainline`: отдельный массив основной линии
- `branches`: отдельный массив ответвлений

Порядок основной прогрессии задаётся самим порядком в `mainline`. Следующий узел mainline не нужно вручную писать в `children`.

Минимальный пример:

```json5
{
  apiVersion: 1,
  worlds: {
    egypt: {
      data: {
        epicTarget: 'epic_egypt',
      },
      map: {
        mode: 'replace',
        reuseOriginalPositions: true,
        mainline: [
          {
            id: 'lvl-1',
            type: 'level',
            appearance: 'normal',
            levels: ['egypt1'],
            title: '1',
          },
          {
            id: 'plant-a',
            type: 'plant',
            template: { type: 'plant' },
            plantReward: 'repeater',
          },
          {
            id: 'lvl-2',
            type: 'level',
            appearance: 'normal',
            levels: ['egypt2'],
            title: '2',
            children: ['branch-2-1'],
          },
        ],
        branches: [
          {
            id: 'branch-2-1',
            type: 'level',
            appearance: 'normal',
            levels: ['egypt20_1'],
            title: '2-1',
            relativePosition: { x: 220, y: 180 },
          },
        ],
      },
    },
  },
}
```

## Что делает `replace`

Когда `map.mode` равно `replace`:

- видимый основной граф полностью задаётся вашими `mainline` и `branches`
- порядок массива `mainline` задаёт прогрессию
- `children` в основном используется для дополнительных боковых связей
- оригинальные vanilla-острова основной карты удаляются
- endless-острова пока остаются vanilla и здесь не задаются

Именно поэтому отдельный массив `mainline` сейчас является предпочтительным вариантом.  
Он устраняет старую проблему, когда вставка узлов растений или наград ломала и порядок прогрессии, и повторное использование позиций.

## Правила позиций

Если у узла не указано `position` явно:

- узлы `mainline` сначала пытаются взять vanilla-позиции по **индексу массива mainline**
- если подходящей оригинальной позиции нет, GP-Next использует авторазмещение

Для узлов веток обычно удобнее `relativePosition`.

## Частые типы узлов

Часто используемые значения `type`:

- `level`
- `plant`
- `giftBox`
- `upgrade`
- `epicPortal`

### `level`

Обычные уровни, боссы и мини-босс острова всё равно остаются `type: 'level'`. Разница в основном во внешнем виде.

Частые значения `appearance`:

- `normal`
- `gargantuar`
- `zomboss`

Например:

- vanilla-остров "маленького босса" на Egypt 8 относится к семейству `gargantuar`
- острова Zomboss на Egypt 25 / 35 используют `zomboss`

Важный момент:

- `appearance` обозначает только широкое визуальное семейство, а не уникальный шаблон
- если в одном мире есть несколько островов одной визуальной группы, но с разной компоновкой, лучше привязывать шаблон к реальному vanilla id уровня

Более надёжные примеры для Egypt:

```json5
template: { levelId: 'egypt8' }
template: { levelId: 'egypt35' }
```

### `giftBox` и `epicPortal`

Сейчас эти два типа уже отображаются корректно, но в отличие от обычного `level` им обычно лучше явно задавать шаблон:

```json5
template: { type: 'giftBox' }
template: { type: 'epicPortal' }
```

Теперь GP-Next ограничивает такие селекторы настоящими vanilla-кандидатами специальных узлов, а не пытается свободно сопоставлять их с посторонними runtime-островами.

## Поле `data` уровня мира

Помимо `map`, можно переопределять некоторые поля уровня мира:

- `levels`
- `plants`
- `startingLevels`
- `epicTarget`
- `intro`

Чаще всего нужны:

- `epicTarget`: в какой эпический мир в итоге ведёт эпический портал
- `intro`: ключ вступления мира

Если в режиме `replace` вы опустите `levels / plants / startingLevels`, GP-Next попытается вывести их из списка узлов.

## Текущие известные ограничения

### 1. Endless-острова пока не стоит настраивать вручную

Пока endless-острова остаются vanilla.

Проблема не только в том, что "не найден шаблон". Внутри endless-острова всё ещё остаются узлами `level`, но также зависят от:

- `EndlessZoneEntrance.levelsID`
- больших шаблонов островов, зависящих от конкретного мира

Если пытаться заменять их как обычные узлы, легко получить:

- наложение нескольких endless-островов
- лишние синие узлы обычных уровней
- endless-входы, прикреплённые не к тому острову

Поэтому текущая рекомендация простая:

- основную карту менять можно
- endless-острова пока лучше оставить vanilla

### 2. Награды `giftBox` пока нельзя точно настраивать здесь

Награды подарочных коробок на карте мира не идут по той же цепочке, что и обычные награды за прохождение уровня.  
То есть это нельзя надёжно контролировать простой правкой level JSON.

Пока не стоит придумывать поля вроде:

```json
{
  "giftReward": "..."
}
```

Сейчас GP-Next не предоставляет точное управление наградами `giftBox` через `gpn-worldmap.json5`.

## Когда использовать `WorldmapFeatures`, а когда `gpn-worldmap`

Практическое правило:

- если вы хотите изменить, какие уровни, растения или epic target есть у мира, смотрите `WorldmapFeatures`
- если вы хотите изменить порядок узлов, структуру mainline/branches, либо расположение коробок и порталов, используйте `gpn-worldmap`

Многие моды карты мира в итоге будут использовать оба подхода:

- один патч для `WorldmapFeatures`
- один патч для `gpn-worldmap`

## Рекомендуемый чек-лист отладки

Если карта отображается не так, как вы ожидали, сначала проверьте:

1. что `worldmap-json` включён во вкладке **Experimental**
2. что имя файла — `gpn-worldmap.json5` или `gpn-worldmap.json`
3. что в корне присутствует `apiVersion: 1`
4. что ключ внутри `worlds` — это правильный `CODENAME` мира
5. если специальный остров выглядит неправильно, попробуйте более точную привязку `template`
